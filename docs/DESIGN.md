# 뉴스 수집 AI 에이전트 설계서

## 1. 목표 및 요구사항

### 1.1 목표
- **AI·증권·금융·IB 업계**의 지속적인 AI 도입·변화를 추적
- 일정 간격(예: 1시간)마다 뉴스 수집
- 중복 뉴스는 병합·수정하여 하나의 품질 높은 항목으로 보관
- Google AI(Gemini) API 활용

### 1.2 관심 키워드/도메인 (설정 가능)
| 분류 | 예시 키워드/주제 |
|------|------------------|
| 증권사 AI/AX/DX | 미래에셋증권, 증권사 AI 도입, AX, DX, 디지털 전환 |
| 글로벌 AI 서비스 | ChatGPT, Claude, Gemini, 컴퓨터 유즈, AI 에이전트 |
| 금융/증권 | Financial AI, 증권사, IB(투자은행), 퀀트, 리테일/기관 |
| 이벤트 | 신규 버전 출시, 신규 서비스, M&A, 규제, 도입 사례 |

### 2.1 핵심 컴포넌트

| 컴포넌트 | 역할 | 기술 후보 |
|----------|------|-----------|
| **설정 관리** | 키워드, 수집 소스, 수집 주기, API 키 등 관리 | YAML/JSON 설정 파일 |
| **스케줄러** | 주기적 실행(예: 1시간) | APScheduler (interval/cron) |
| **수집기** | RSS, 뉴스 API에서 기사 목록/본문 수집 | feedparser, requests, GNews/NewsAPI 등 |
| **정규화** | 제목/URL/날짜/출처 통일 형식 | Python 데이터 클래스 |

| **Gemini 연동** | 요약, 병합 요약, 키워드 추출 | Google GenAI SDK (`google-genai`) |
| **저장소** | 기사·요약·메타데이터 영구 저장 | SQLite (단일 파일, 이력 관리 용이) |

## 3. 데이터 모델

### 3.1 기사(Article) 스키마

| 필드 | 타입 | 설명 |
|------|------|------|
| id | str (UUID) | 고유 식별자 |
| url | str | 원문 URL (중복 판단 1차 키) |
| url_hash | str | URL 해시 (인덱스용) |
| title | str | 제목 |
| summary | str | Gemini로 생성한 요약 |
| body_snippet | str | 본문 일부(저장용, 선택) |
| source | str | 출처(도메인 또는 매체명) |
| published_at | datetime | 게시 시각 |
| collected_at | datetime | 수집 시각 |
| keywords | list[str] | 태그/키워드 |
| category | str | AI/증권/금융/IB 등 분류 |
| duplicate_group_id | str? | 동일 뉴스 묶음 ID (중복 시) |
| version | int | 동일 url에 대한 갱신 횟수(수정 시 증가) |

### 3.2 중복 그룹(DuplicateGroup) 및 통합 정보 스키마

**유사 뉴스(같은 사건)** 는 하나로 묶어 **자동 통합**하고, **가장 많은 정보를 담은 하나의 통합 요약**으로 만든다.

| 필드 | 타입 | 설명 |
|------|------|------|
| id | str | 그룹 ID (UUID) |
| canonical_article_id | str | 대표 기사 ID (목록/노출용) |
| merged_summary | str | **통합 요약** — 그룹 내 모든 기사의 정보를 최대한 반영해 Gemini가 생성 |
| merged_title | str? | 통합 제목 (선택, 대표 제목 또는 AI 생성) |
| source_article_ids | list[str] | 이 그룹에 포함된 모든 기사 ID |
| source_urls | list[str] | 원본 기사 URL 목록 (링크 모음용) |
| created_at | datetime | 그룹 최초 생성 시각 |
| updated_at | datetime | 마지막 병합/수정 시각 |
| merge_status | str | `auto` / `manual` / `edited` (통합 요약 수동 수정 여부) |

**통합 원칙**: 병합 시 "중복 사실은 한 번만"이 아니라 **서로 다른 관점·수치·출처 정보를 모두 포함**해 가장 많은 정보가 담긴 하나의 요약으로 생성(재병합 시에도 동일).

**통합 정보 활용**: 목록에서는 그룹 단위로 한 줄 표시(통합 제목 + 통합 요약). 상세에서 "같은 뉴스로 묶인 기사" 목록·원문 링크 표시. 새 기사가 기존 그룹과 중복이면 그룹에 추가 후 `merged_summary` 재생성(재병합).

### 3.3 연관 뉴스 히스토리(RelatedChain) 스키마

**유사 뉴스가 아닌 새 뉴스**이지만 **이전 뉴스와 연관**된 경우(후속 보도, 같은 주제·엔티티) **히스토리(체인)** 형태로 연결해, 사용자가 이전/이후 보도를 따라갈 수 있게 한다.

| 필드 | 타입 | 설명 |
|------|------|------|
| id | str | 체인 ID (UUID) |
| article_ids | list[str] | 연관 기사 ID 목록. **시간순 정렬**(published_at 기준, 오래된 것 → 최신). 그룹인 경우 대표 기사(canonical) ID만 넣거나, 그룹 id를 별도 필드로 |
| topic_label | str? | 주제 라벨 (예: "미래에셋 AI 도입", "ChatGPT API") — UI에서 히스토리 제목으로 사용 가능 |
| created_at | datetime | 체인 최초 생성 시각 |
| updated_at | datetime | 마지막 기사 추가 시각 |

**연관 판단 기준(자동)**: 키워드·엔티티 겹침, 또는 Gemini 질의("이 기사가 다음 중 어떤 기사의 후속/연관인가?")로 기존 기사와 연결. **중복 그룹에 속한 기사는 같은 사건이므로 체인에는 대표 1건만** 포함하고, **서로 다른 사건이지만 같은 흐름(예: 같은 회사 AI 도입 1차→2차)** 이면 체인에 순서대로 추가.

**UI 활용**: 기사/그룹 상세 화면에서 "관련 뉴스 히스토리" 섹션 — **이전 보도** / **후속 보도** 링크 또는 **타임라인**으로 목록 표시. 클릭 시 해당 기사/그룹 상세로 이동하며, 같은 체인 내에서 앞뒤로 탐색 가능.

## 4. 처리 파이프라인 상세

### 4.1 수집 단계
1. 설정에서 **키워드 목록**과 **RSS/API 소스 목록** 로드
2. 각 소스별로 요청 (GNews Search, NewsAPI, RSS 등)
   - 키워드: "증권사 AI", "ChatGPT", "Claude", "미래에셋증권", "Financial AI", "IB AI" 등
3. 응답을 공통 **Article** 구조로 정규화 (제목, URL, 본문 일부, 출처, 날짜)

### 4.2 중복 탐지 전략 (실용적 2단계)
1. **1단계: URL/정규화 URL**
   - URL 정규화(쿼리 제거, 트래킹 파라미터 제거) 후 해시
   - 동일 URL → 기존 레코드 갱신(본문/요약 개선 시만 업데이트)
2. **2단계: 제목 유사도**
   - 새 기사 제목과 DB 내 최근 N일 제목 비교
   - 유사도: 정규화 후 Jaccard/편집거리 또는 **Gemini에 “이 두 제목이 같은 사건인가?” 질의**
   - 임베딩 사용 시: sentence-transformers 또는 Gemini embedding API로 벡터 저장 후 코사인 유사도(예: 0.92 이상 → 중복 후보)
3. **3단계: 중복일 경우 하나의 통합 정보로 만들기**
   - 중복 후보가 **기존 그룹**의 기사와 유사하면 → 해당 그룹에 새 기사 추가
   - 중복 후보끼리만 있으면 → **새 DuplicateGroup 생성**: canonical_article_id 1개, 나머지는 source_article_ids·source_urls에 포함
   - 그룹에 대해 **Gemini 병합 요약** 1회: 그룹 내 **모든 기사 제목·요약·본문 일부**를 넣고 **"가장 많은 정보를 담은 하나의 통합 요약으로 정리. 중복된 사실은 한 번만, 서로 다른 관점·수치·출처·세부 내용은 빠짐없이 포함"** → `merged_summary` 저장. 재병합 시에도 동일 원칙.
   - 그룹에 새 기사가 추가될 때마다 **재병합**: `merged_summary`·`updated_at` 갱신
   - 목록/웹에서는 그룹 단위로 "통합 제목 + 통합 요약 + 원문 링크 N개"로 노출

### 4.2b 연관 뉴스(히스토리) 연결 — 새 뉴스이지만 이전 뉴스와 연관
**유사 뉴스가 아닌 새 뉴스**이지만 **이전 뉴스와 연관**된 경우(후속 보도, 같은 주제·엔티티) **히스토리 체인**으로 연결한다.
- **연관 탐지**: 새 기사(또는 그룹 대표) 저장 후, 키워드·엔티티 겹침으로 후보 추린 뒤 **Gemini에 "이 기사가 아래 기사들 중 어떤 것의 후속/연관인가?"** 질의해 연관 여부·순서 판단. 또는 키워드/카테고리 유사 + 시간순으로 같은 주제 체인에 추가.
- **체인 생성/갱신**: 연관이 있으면 기존 **RelatedChain**에 해당 기사(또는 그룹 대표 id)를 **시간순(published_at)** 으로 삽입. 없으면 새 RelatedChain 생성.
- **저장**: `related_chains` 테이블에 id, article_ids(순서 유지), topic_label, updated_at 저장.
- **UI**: 기사/그룹 상세에서 **"관련 뉴스 히스토리"** — 이전 보도 / 후속 보도 목록 또는 타임라인으로 표시, 클릭 시 해당 항목으로 이동하며 체인을 따라 탐색 가능하게 구현.

### 4.3 Gemini 활용
- **요약 생성**: 수집된 본문(또는 snippet) → “3문장 이내 요약, 핵심 수치/날짜 유지”
- **병합 요약**: 중복 그룹에 속한 여러 기사 내용 → “하나의 통합 요약으로 정리”
- **분류/키워드**: “이 기사를 AI/증권/금융/IB 중 분류하고 키워드 3~5개 추출”
- **중복 판단(선택)**: “다음 두 제목이 같은 사건/뉴스를 다루는지 Yes/No와 이유 한 줄”

### 4.4 저장/갱신 규칙
- **신규 URL**: 새 Article 삽입, `duplicate_group_id` = null(단독) 또는 그룹 id
- **기존 URL**: `version` 증가, `summary`/`body_snippet` 등 갱신, `collected_at` 갱신
- **유사 제목으로 그룹 판단된 경우**: 기존 그룹에 추가하거나 새 그룹 생성 → **통합 요약(merged_summary) 생성/재생성** 후 저장

## 4.5 가능한 빠짐 없이 수정할 수 있는 방법

설정·데이터를 **어디서든 수정 가능**하게 하여 누락 없이 유지보수할 수 있도록 한다.

### 4.5.1 설정 수정
- **config.yaml**: 키워드, 소스(RSS/API URL), 수집 주기, 중복 임계값, 모델명 등 **모든 수집/파이프라인 설정**을 한 곳에서 수정
- **적용 방식**: 스케줄러 기동 시 config 로드; 필요 시 **설정 변경 후 재시작** 또는 **hot-reload**(config 파일 변경 감지 시 다음 수집 주기부터 새 설정 적용) 지원
- **웹에서 설정 편집**: 선택 사항으로, 키워드 추가/삭제·소스 on/off를 웹 UI에서 수정하고 config.yaml에 반영(또는 DB에 설정 저장)하도록 구현

### 4.5.2 키워드·소스 추가/삭제
- 키워드: `config.sources[].keywords` 리스트에 항목 추가/제거만 하면 다음 수집부터 반영
- 소스: `sources`에 새 `type`(예: rss, gnews) 항목 추가 시 해당 수집기가 동작하도록 팩토리에서 처리
- **빠짐 방지**: 문서에 "추가하고 싶은 키워드/소스는 config.yaml의 어느 섹션을 수정하면 되는지" 명시

### 4.5.3 기사·통합 요약 수동 수정
- **개별 기사**: 제목, 요약(summary), 키워드, category를 **웹 UI 또는 API**에서 수정 가능하게 하고, 저장소에 반영 시 `version` 증가
- **통합 요약(merged_summary)**: 그룹 상세 화면에서 "통합 요약 편집" 버튼으로 텍스트 수정 가능; 저장 시 `merge_status` = `edited`로 두어 자동 재병합 시 덮어쓸지 선택 가능하게 함
- **그룹 해제/재편**: 두 기사를 "다른 뉴스"로 나누거나, 별도 기사를 기존 그룹에 넣는 **수동 병합/분리** 기능(웹 또는 API) 제공

### 4.5.4 수정 이력(선택)
- 기사/그룹의 **수정 이력**을 별도 테이블(article_edits, group_edits)에 저장하면, 누가 언제 무엇을 바꿨는지 추적 가능
- 초기에는 `version`·`updated_at`만으로도 "마지막 수정 시점" 파악 가능

## 5. 설정 설계 (config)

```yaml
# config.yaml 예시
schedule:
  interval_minutes: 60   # 수집 주기(분)

sources:
  # API (키 필요 시 env에서 로드)
  - type: gnews
    api_key_env: GNEWS_API_KEY
    keywords:
      - "증권사 AI"
      - "미래에셋증권 AI"
      - "ChatGPT"
      - "Claude AI"
      - "Financial AI"
      - "investment bank AI"
  - type: rss
    url: "https://example.com/ai-news.xml"

google_ai:
  api_key_env: GOOGLE_AI_API_KEY
  model: "gemini-2.0-flash"  # 요약/분류용

storage:
  path: "./data/news.db"   # SQLite 경로

# 웹 접근 제어 (비밀번호로 뉴스 페이지 접근)
web:
  password_env: WEB_PASSWORD   # 환경변수에 비밀번호 설정 (예: .env)
  session_ttl_hours: 168       # 세션 유효 시간(시간), 기본 7일. 다중 기기 동시 접속 허용

dedup:
  title_similarity_threshold: 0.88
  use_embedding: false     # true 시 Gemini embedding 또는 로컬 모델
```

## 9. 웹 페이지를 통한 조회·수정

뉴스와 통합 정보를 **웹에서 쉽게 보고**, 필요 시 **수정**할 수 있도록 한다.

### 9.1 제공 화면
| 화면 | 기능 |
|------|------|
| **목록(피드)** | 그룹 단위로 카드/리스트 표시. 그룹이면 "통합 제목 + 통합 요약 + 원문 N개 링크"; 단독 기사면 제목+요약+원문 1개. 날짜순/카테고리별 정렬. |
| **필터** | 카테고리(AI/증권/금융/IB), 키워드 태그, 기간, 출처(source)로 필터링. |
| **검색** | 제목·요약·키워드 전체 검색(LIKE 또는 간단 FTS). |
| **상세(기사)** | 단일 기사: 제목, 요약, 본문 일부, 원문 링크, 키워드, 카테고리, 수집/수정 시각. **편집** 버튼 → 제목/요약/키워드/카테고리 수정 폼. **관련 뉴스 히스토리** — 이전 보도/후속 보도 링크 또는 타임라인으로 표시, 클릭 시 해당 기사로 이동하며 체인 따라가기. |
| **상세(그룹)** | 통합 제목, **통합 요약**(편집 가능), "같은 뉴스로 묶인 기사" 목록(각각 제목, 출처, 링크). 그룹에서 기사 제외/추가(수동 병합·분리). **통합 요약 편집** 시 merge_status = edited. **관련 뉴스 히스토리** — 동일하게 이전/후속 보도 따라가기. |
| **설정(선택)** | 키워드 추가/삭제, RSS/API 소스 on/off, 수집 주기. config.yaml 반영 또는 DB 설정 저장. |

### 9.2 기술 구현
- **백엔드**: FastAPI. GET 목록/상세/검색·필터, PATCH 기사/그룹 수정, (선택) 설정 조회·수정.
- **프론트**: 정적 HTML + Alpine.js 또는 Vanilla JS, 또는 React/Vue 등 SPA. 서버에서 JSON 내려주고 클라이언트에서 렌더링. **상세는 [FRONTEND_DESIGN.md](./FRONTEND_DESIGN.md) 참고.**
- **실행**: `uvicorn`으로 웹 서버 기동; 스케줄러(main)와 같은 프로세스에서 백그라운드로 돌리거나, 별도 프로세스로 분리 가능.

### 9.3 접근 제어(비밀번호·다중 기기)
- **비밀번호 인증**: 뉴스 조회 페이지 접속 시 **비밀번호 입력** 후에만 목록/상세/수정 화면 접근 가능. 비밀번호는 config 또는 환경변수(`WEB_PASSWORD`)로 설정.
- **세션 기반·다중 기기 동시 접속**: 로그인 시 **세션 쿠키**(또는 세션 ID) 발급. **다른 기기에서 로그인해도 이전 기기의 세션은 유지**되며 끊기지 않음(동시 다중 세션 허용). 세션 저장은 서버 메모리 또는 Redis 등으로 관리.
- **보호 범위**: `/`(피드), `/articles/*`, `/groups/*`, `/api/*` 중 뉴스 데이터를 반환하는 경로는 모두 인증 필요. 로그인 페이지(`/login`)와 로그인 API(`POST /api/auth/login`)는 인증 없이 접근 가능.

### 9.4 Docker·포트
- **전체 서비스 Docker 컨테이너화**: 앱(웹 + 스케줄러), 데이터(volume 마운트), 설정·env는 모두 **Docker 컨테이너**로 실행.
- **외부 접근 포트**: 웹 서버는 **6800** 포트로 노출. 호스트에서 `http://<서버>:6800`으로 접속.
- **구성**: `Dockerfile`로 이미지 빌드, `docker-compose.yml`로 컨테이너 기동·볼륨·포트 6800:6800(또는 80→6800) 매핑. SQLite 데이터는 volume으로 영속화.

### 9.5 디렉터리 반영
- `src/web/` 또는 `web/`: FastAPI 앱, 라우터(목록/상세/검색/수정), **인증 미들웨어·로그인 라우트**, 템플릿 또는 정적 파일.
- **Docker**: 프로젝트 루트에 `Dockerfile`, `docker-compose.yml`, (선택) `.dockerignore`.
- API와 페이지 경로: 예) `/login`, `/`, `/articles`, `/articles/{id}`, `/groups/{id}`, `/api/auth/login`, `/api/feed` 등.

이 설계를 바탕으로 **단계별 프로그래밍 계획**에서 웹 Phase를 포함해 구현 순서를 정리합니다.
