# 프론트엔드 상세 설계

이 문서는 뉴스 수집 AI 에이전트의 **웹 프론트엔드** 구성을 상세히 정의한다. [DESIGN.md](./DESIGN.md) §9 및 [PROGRAMMING_PLAN.md](./PROGRAMMING_PLAN.md) Phase 6과 연동한다.

---

## 1. 기술 스택 및 선택 근거

### 1.1 권장: 서버 렌더링(HTML) + Alpine.js

| 구분 | 선택 | 이유 |
|------|------|------|
| **렌더링** | 서버에서 HTML 생성(Jinja2 등) + 클라이언트에서 일부만 보강 | SEO 불필요, 비밀번호 보호만 필요. 초기 로딩 단순, 빌드 도구 최소화 |
| **인터랙션** | Alpine.js | 가벼움(빌드 없이 CDN), 필터/검색/모달/편집 폼 등에 충분. 학습 부담 적음 |
| **스타일** | CSS (단일 stylesheet 또는 Tailwind CDN) | 컴포넌트 수가 많지 않아 단일 CSS 또는 유틸리티 클래스로 충분 |
| **API 호출** | fetch + credentials: 'include' | 세션 쿠키 자동 전송. 401 시 `/login`으로 리다이렉트 |

**대안**
- **Vanilla JS**: Alpine 대신 순수 JS로 필터/검색/모달 구현 가능. 코드량 다소 증가.
- **React/Vue SPA**: 복잡한 편집·히스토리 UI 확장 시 고려. 별도 빌드·라우터 필요.

이하 설계는 **서버 렌더링 + Alpine.js** 기준으로 서술한다.

### 1.2 배치

- **템플릿**: `src/web/templates/` — Jinja2 HTML. 레이아웃 1개, 페이지별 상속.
- **정적 자산**: `src/web/static/` — `css/app.css`, `js/app.js`(Alpine 초기화·공통 함수), (선택) `img/`.
- **Alpine**: CDN으로 `index.html` 또는 레이아웃에 `<script defer src="https://cdn.jsdelivr.net/npm/alpinejs@3.x.x/dist/cdn.min.js"></script>`.

## 2. 라우팅 및 URL 구조

| URL | 설명 | 인증 | 메서드 |
|-----|------|------|--------|
| `/login` | 로그인 폼 | 불필요 | GET |
| `/` | 피드(목록) | 필요 | GET |
| `/search` | 검색 결과 (쿼리 `q`) | 필요 | GET |
| `/articles/<id>` | 기사 상세 | 필요 | GET |
| `/groups/<id>` | 그룹(통합 뉴스) 상세 | 필요 | GET |
| `/api/auth/login` | 로그인 | 불필요 | POST |
| `/api/auth/logout` | 로그아웃 | 필요 | POST |
| `/api/feed` | 피드 데이터 | 필요 | GET |
| `/api/search` | 검색 | 필요 | GET |
| `/api/articles/<id>` | 기사 상세/수정 | 필요 | GET, PATCH |
| `/api/groups/<id>` | 그룹 상세/수정 | 필요 | GET, PATCH |
| `/api/articles/<id>/related-history` | 관련 뉴스 히스토리 | 필요 | GET |
| `/api/groups/<id>/related-history` | 그룹 기준 관련 히스토리 | 필요 | GET |

- **클라이언트 라우팅**: 없음. 모든 페이지는 서버가 HTML 반환(같은 base template 사용). 필터/검색은 쿼리 변경 후 같은 `/` 또는 `/search` 재요청 또는 fetch로 데이터만 갱신(선택).

## 3. 공통 레이아웃

### 3.1 레이아웃 구조 (base.html)

```
┌─────────────────────────────────────────────────────────────────┐
│  헤더: [로고/타이틀]  [검색창]                    [로그아웃]     │
├─────────────────────────────────────────────────────────────────┤
│  (선택) 필터 바: [카테고리] [키워드] [기간] [출처] [적용]        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  메인 콘텐츠 (페이지별 블록)                                      │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│  푸터: 간단 문구 또는 생략                                        │
└─────────────────────────────────────────────────────────────────┘
```

- **헤더**: 모든 인증된 페이지에서 공통. 검색창은 입력 후 Enter 또는 버튼 클릭 시 `/search?q=...`로 이동. 로그아웃은 버튼 클릭 시 `POST /api/auth/logout` 후 `/login`으로 이동.
- **필터 바**: 피드(`/`)에서만 표시. 카테고리(AI/증권/금융/IB), 키워드 태그(다중 선택 가능), 기간(from/to), 출처(source) 드롭다운. "적용" 시 현재 URL 쿼리 변경 후 피드 데이터 다시 로드(fetch 또는 form GET).
- **로그인 페이지**는 레이아웃 없이 단순 폼만 사용 가능.

### 3.2 인증 처리

- **페이지 로드 시**: 서버에서 세션 검사. 미인증이면 `302 /login` 또는 클라이언트에서 API 호출 401 수신 시 `window.location.href = '/login'`.
- **API 호출**: `fetch(url, { credentials: 'include' })`. 401 응답 시 전역에서 `location.href = '/login'` 처리.
- **로그인 성공 후**: `POST /api/auth/login` 성공 시 `location.href = '/'` 또는 `location.href = next`(쿼리로 전달된 return URL).

## 4. 페이지별 상세 설계

### 4.1 로그인 페이지 (`/login`)

| 항목 | 내용 |
|------|------|
| **구성** | 제목("뉴스 에이전트" 등), 비밀번호 입력 필드 1개, "로그인" 버튼, (선택) 에러 메시지 영역 |
| **동작** | 폼 submit 시 `POST /api/auth/login`, body: `{ "password": "..." }`. 성공 시 200 + 세션 쿠키 설정 → 리다이렉트 `/`. 실패 시 4xx + 메시지 → 화면에 "비밀번호가 올바르지 않습니다" 등 표시 |
| **레이아웃** | 중앙 정렬 카드, 배경은 단색 또는 미니멀 패턴. 헤더/푸터 없음 |
| **접근성** | label for + id, type="password", autocomplete="current-password" |

---

### 4.2 피드(목록) 페이지 (`/`)

#### 4.2.1 데이터

- **API**: `GET /api/feed?category=&keyword=&source=&since=&until=&limit=20&offset=0`
- **응답**: `{ "items": [ ... ], "total": N }`. 각 item은 **그룹** 또는 **단독 기사**.
  - 그룹: `type: "group", id, merged_title, merged_summary, source_urls: [{ url, title?, source }], published_at(대표 시각), category, keywords`
  - 단독: `type: "article", id, title, summary, url, source, published_at, category, keywords`
- **표시 순서**: 날짜(published_at) 내림차순.

#### 4.2.2 레이아웃

- **상단**: 필터 바(카테고리, 키워드, 기간, 출처). Alpine로 값 보관, "적용" 클릭 시 쿼리 반영 후 `fetch('/api/feed?' + params)` 호출해 `items` 갱신.
- **목록**: 카드 리스트. 그룹 카드와 기사 카드가 동일한 카드 스타일로 섞여 표시.

#### 4.2.3 카드 컴포넌트(공통)

- **그룹 카드**
  - **제목**: `merged_title` (없으면 대표 기사 제목). 클릭 시 `/groups/<id>` 이동.
  - **요약**: `merged_summary` 2~3줄 말줄임.
  - **메타**: 날짜, 카테고리, 키워드 태그.
  - **원문 링크**: "원문 N개" 또는 작은 버튼들 — 클릭 시 `source_urls[i].url` 새 탭에서 열기.
- **기사 카드**
  - **제목**: `title`. 클릭 시 `/articles/<id>` 이동.
  - **요약**: `summary` 2~3줄 말줄임.
  - **메타**: 날짜, 출처, 카테고리, 키워드.
  - **원문**: "원문 보기" 1개 링크.

#### 4.2.4 페이지네이션 / 더 보기

- **방식 1**: "더 보기" 버튼. 클릭 시 `offset += limit`으로 같은 API 재호출 후 기존 리스트 아래에 append.
- **방식 2**: 페이지 번호 (1, 2, 3…). `offset = (page - 1) * limit`으로 요청 후 목록 전체 교체.

---

### 4.3 검색 페이지 (`/search?q=...`)

| 항목 | 내용 |
|------|------|
| **진입** | 헤더 검색창에서 입력 후 Enter 또는 검색 버튼 → `/search?q=검색어` |
| **API** | `GET /api/search?q=검색어&limit=20&offset=0` |
| **응답** | 피드와 동일한 형태의 `items` (그룹/기사 혼합). 검색 매칭된 항목만 |
| **표시** | 피드와 동일한 카드 리스트. "검색어: xxx" 제목 표시, 결과 0건 시 "검색 결과가 없습니다" 메시지 |
| **페이지네이션** | 피드와 동일하게 "더 보기" 또는 페이지 번호 |

---

### 4.4 기사 상세 페이지 (`/articles/<id>`)

#### 4.4.1 데이터

- **API**: `GET /api/articles/<id>`
- **응답**: 기사 필드 전체 + (있을 경우) `group` 정보 + **`related_history`**
  - `related_history`: `{ "chains": [ { "chain_id", "topic_label", "items": [ { "article_id" or "group_id", "title", "published_at", "position": "prev"|"current"|"next" } ] } ] }`. 현재 기사가 체인 내 어디에 있는지, 이전/다음 노드 포함.

#### 4.4.2 레이아웃

```
┌────────────────────────────────────────────────────────────┐
│  [목록으로] (또는 브레드크럼: 피드 > 기사)                   │
├────────────────────────────────────────────────────────────┤
│  제목 (h1)                                                 │
│  메타: 날짜, 출처, 카테고리, 키워드 태그                     │
├────────────────────────────────────────────────────────────┤
│  요약 (본문 영역)                                           │
│  body_snippet (있으면)                                     │
├────────────────────────────────────────────────────────────┤
│  [원문 보기] (외부 링크)                                    │
├────────────────────────────────────────────────────────────┤
│  관련 뉴스 히스토리 (아래 4.4.3 참고)                       │
├────────────────────────────────────────────────────────────┤
│  [편집] 버튼 → 클릭 시 편집 폼 표시 (아래 4.4.4 참고)       │
└────────────────────────────────────────────────────────────┘
```

#### 4.4.3 관련 뉴스 히스토리 UI

- **목적**: 이 기사가 속한 "연관 뉴스 체인"에서 **이전 보도 / 후속 보도**를 따라갈 수 있게 함.
- **표시 방식(택일 또는 병행)**  
  - **방식 A — 이전/다음 링크**: "이전 보도: [제목]" / "후속 보도: [제목]" 한 줄씩. 클릭 시 `/articles/<id>` 또는 `/groups/<id>`로 이동.  
  - **방식 B — 타임라인**: 세로 타임라인. 각 노드에 제목·날짜, 현재 기사는 강조. 클릭 시 해당 상세로 이동.
- **체인이 여러 개인 경우**: "주제: topic_label" 별로 섹션 나누어 표시. 각 섹션에 이전/다음 또는 타임라인.
- **데이터**: 위 `related_history.chains`를 그대로 사용. `position === "current"`인 항목은 시각적으로 현재 페이지로 표시(비활성 또는 굵게).

#### 4.4.4 기사 편집

- **트리거**: "편집" 버튼 클릭.
- **UI**: 인라인 폼 또는 모달. 필드: 제목, 요약, 키워드(쉼표 또는 태그 입력), 카테고리(드롭다운).
- **저장**: `PATCH /api/articles/<id>`, body: `{ "title", "summary", "keywords", "category" }`. 성공 시 상세 데이터 다시 로드하여 화면 갱신. 실패 시 에러 메시지 표시.
- **취소**: 폼 닫기, 원래 읽기 모드로 복귀.

---

### 4.5 그룹 상세 페이지 (`/groups/<id>`)

#### 4.5.1 데이터

- **API**: `GET /api/groups/<id>`
- **응답**: 그룹 필드 전체(`merged_title`, `merged_summary`, `source_urls`, `source_article_ids` 등) + 각 source 기사 요약 정보(title, source, url) + **`related_history`**(기사 상세와 동일 구조, 대표 기사 기준).

#### 4.5.2 레이아웃

```
┌────────────────────────────────────────────────────────────┐
│  [목록으로] (또는 브레드크럼: 피드 > 통합 뉴스)              │
├────────────────────────────────────────────────────────────┤
│  통합 제목 (h1)                                            │
│  메타: 대표 날짜, 카테고리, 키워드                           │
├────────────────────────────────────────────────────────────┤
│  통합 요약 (본문 영역)                    [통합 요약 편집]  │
├────────────────────────────────────────────────────────────┤
│  같은 뉴스로 묶인 기사 (N건)                                │
│  - [기사1 제목] (출처) [원문]                               │
│  - [기사2 제목] (출처) [원문]                               │
├────────────────────────────────────────────────────────────┤
│  관련 뉴스 히스토리 (기사 상세와 동일한 UI)                  │
└────────────────────────────────────────────────────────────┘
```

- **같은 뉴스로 묶인 기사**: 리스트로 제목, 출처, 원문 링크. 원문은 새 탭.
- **관련 뉴스 히스토리**: 기사 상세와 동일하게 이전 보도/후속 보도 또는 타임라인. 클릭 시 해당 `/articles/<id>` 또는 `/groups/<id>`로 이동.

#### 4.5.3 통합 요약 편집

- **트리거**: "통합 요약 편집" 버튼.
- **UI**: 요약 영역을 textarea로 전환(또는 모달). 저장 시 `PATCH /api/groups/<id>`, body: `{ "merged_summary": "..." }`. 성공 시 화면 갱신. 서버에서 `merge_status` = `edited` 반영.

## 5. API 연동 공통 사항

### 5.1 fetch 래퍼

- **baseURL**: 상대 경로 `/api` (같은 오리진).
- **options**: `credentials: 'include'`, `headers: { 'Content-Type': 'application/json' }` (POST/PATCH 시).
- **401 처리**: `if (res.status === 401) { window.location.href = '/login'; return; }`.
- **4xx/5xx**: 응답 body의 메시지가 있으면 표시, 없으면 "요청을 처리할 수 없습니다" 등 공통 메시지.

### 5.2 로딩·에러 상태

- **로딩**: 목록/상세 로드 시 스피너 또는 스켈레톤 표시. Alpine이면 `x-data="{ loading: true }"` → fetch 완료 후 `loading = false`.
- **에러**: 목록/검색 실패 시 "목록을 불러올 수 없습니다" + 재시도 버튼. 상세 실패 시 "기사를 찾을 수 없습니다" + 목록으로 링크.

### 5.3 날짜 표시

- **서버**: ISO 8601 문자열로 전달.
- **클라이언트**: `published_at`을 로컬 시간으로 포맷 (예: "2025-03-12 14:30", 또는 "2시간 전"). 간단히 `new Date(s).toLocaleString()` 또는 작은 라이브러리 사용.

## 6. 반응형·접근성

- **반응형**: 목록은 1열(모바일) ~ 2열(태블릿) ~ 3열(데스크톱) 그리드. 카드 너비는 max-width 제한. 필터 바는 작은 화면에서 접이식(드롭다운) 가능.
- **접근성**: 버튼/링크에 적절한 label, 제목 계층(h1 → h2) 준수, 폼에 label-for, 에러 메시지는 role="alert" 또는 aria-live.

## 7. 디렉터리 구조(프론트 기준)

```
src/web/
├── app.py                 # FastAPI 앱, 정적/템플릿 마운트
├── routes.py              # HTML 페이지 라우트 + API 라우트
├── templates/
│   ├── base.html          # 공통 레이아웃 (헤더, 필터 바, 블록)
│   ├── login.html
│   ├── feed.html          # 피드(목록)
│   ├── search.html       # 검색 결과 (feed.html과 거의 동일, 제목만 다름)
│   ├── article_detail.html
│   └── group_detail.html
└── static/
    ├── css/
    │   └── app.css
    └── js/
        └── app.js         # fetch 래퍼, 401 처리, (선택) Alpine 데이터 믹스
```

- **정적 파일**: FastAPI에서 `app.mount("/static", StaticFiles(directory="..."))` 로 서빙.
- **템플릿**: Jinja2. 블록 `content`, `title` 등. feed/search는 같은 템플릿에 `is_search` 플래그로 분기 가능.

## 8. 구현 순서 제안

1. **base.html** + 공통 CSS, 헤더(검색창, 로그아웃), 푸터.
2. **login.html** + 로그인 폼 및 `POST /api/auth/login` 연동, 401 시 리다이렉트 처리.
3. **feed.html** + 피드 API 연동, 그룹/기사 카드 렌더링, 필터 바(쿼리 → API 파라미터).
4. **search.html** (또는 feed와 통합) + `/api/search` 연동.
5. **article_detail.html** + 기사 API, 편집 폼, **관련 뉴스 히스토리** (이전/다음 링크부터).
6. **group_detail.html** + 그룹 API, 통합 요약 편집, 같은 뉴스 묶음 목록, **관련 뉴스 히스토리**.
7. 타임라인 UI(선택), 페이지네이션/더 보기, 로딩·에러 상태 마무리.

이 순서로 구현하면 [PROGRAMMING_PLAN.md](./PROGRAMMING_PLAN.md) Phase 6의 프론트 항목을 충족할 수 있다.
